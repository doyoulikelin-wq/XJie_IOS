# XJie iOS Repository Rules

These rules supplement `/Users/linlin/Desktop/X/AGENTS.md` and are mandatory for every change in this repository.

## Before editing behavior

1. Read the workspace memory and `/Users/linlin/Desktop/X/XJie_IOS/docs/quality/REGRESSION_POLICY.md`.
2. Search existing resolved issues, risks, regression contracts, production code, and tests for the same class of behavior.
3. Update `quality/change_impact.json` with the real root cause or risk hypothesis, all affected domains, sibling-entry scan, invariant IDs, tests, manual matrix, and remaining risk.
4. For a bug fix or user-visible behavior change, add or strengthen a named regression test. The test must contain a meaningful assertion or interaction and must be staged with the production change.

## Hard gates

- Run `/usr/bin/python3 -I tools/regression_guard.py validate` before and after editing.
- Run `/usr/bin/python3 -I tools/regression_guard.py check --working` before describing a behavior change as complete.
- Run `/usr/bin/python3 -I tools/run_regression_gate.py impacted` for the domains in `quality/change_impact.json`.
- Never use `git commit --no-verify` or `git push --no-verify`. The tracked hooks are part of the delivery contract.
- Candidate-snapshot hooks must preserve the caller's index only long enough to create the immutable candidate, then clear every name returned by `git rev-parse --local-env-vars` before linked-worktree add/remove and validation. The tools suite must execute the real pre-commit hook with a relative `GIT_INDEX_FILE`; a hook that passes only when called manually is broken and blocks delivery.
- A failing or skipped required gate blocks commit, completion language, deployment, and release.
- A required failure remains blocking even if a later rerun passes. Close it only after preserving the failing evidence, reproducing or otherwise proving the trigger, identifying the root cause, scanning sibling paths, adding a deterministic invariant and meaningful regression assertion, and passing the complete gates on a new exact commit. Never classify a same-tree red/green sequence as harmless flakiness merely because the last run is green.
- CI `quality-gate` must be green on the exact commit. A green job obtained through `|| true`, fail-open `&&`, swallowed pipeline failures, skipped required tests, or a different SHA is invalid.
- Every mandatory `xcodebuild test` command must write an `.xcresult` and pass `tools/validate_xcresult.py`; exit code zero alone is not evidence. The tracked exact inventories are iOS Unit `149`, full UI `5`, small-screen UI `2`, and the Unit/full-UI union `154`. Missing, extra, duplicate, renamed, skipped, failed, or expected-failure IDs are gate failures; the small-screen result must also prove the device is iPhone SE (3rd generation).
- The tracked Python runtime inventories are backend full `264` IDs and tools `74` IDs. Tools permit no skips. Backend full permits only the three explicitly pinned integration placeholders, so its honest result is `261 passed + 3 skipped`, not 264 implemented integration tests; changing an ID, parameter case, skip reason, or allowlist requires an explicit contract and regression-test update.
- A local backend gate may use only `backend/.venv/bin/python` after it resolves to a native executable outside the repository and passes the isolated prefix/purelib probe. The gate binds that executable and every site-packages byte before and after the run, deletes the pinned JUnit output first, and independently re-parses the newly generated exact IDs/skips. Exit code zero, a script launcher, stale XML, or the inner validator alone is never sufficient evidence.
- Application startup may not call `create_all` or execute handwritten schema DDL. The tracked production deployer never runs `alembic upgrade`: it may cut over only when immutable old/candidate images expose byte-identical linear migration histories and canonical SQLAlchemy model manifests and the live database `current` exactly equals that single head. Any migration or model delta is a blocked, separate migration project requiring a trusted PostgreSQL baseline, expand/contract review, old/new CRUD probes, backup/recovery evidence, and its own protected change; a container rollback is not a database rollback.
- Production deployment may be invoked only through the root-controlled `/usr/local/sbin/xjie-production-launch`; `/usr/local/sbin/xjie-production-deploy` is its supervised internal child, not an operator entrypoint. The transactional trust bundle contains exactly seven root-owned, single-link files: launcher/deployer/installer at mode `0555`, plus the container spec, deploy guard, release gate and expected Python inventory under `/usr/local/libexec/xjie-production-deploy` at mode `0444`. All seven exact-SHA candidate files must byte-match the independently approved installed bundle. Before any repository Python runs, any production env path is opened, or any container is deleted, the installed release gate must prove official `main`, the merged PR, exact push check, and the distinct main/XAGE protections using a GitHub token received only through the launcher's anonymous stdin pipe. Bundle replacement is a separate root action permitted only while production is healthy and no cutover journal exists.
- Before a no-delta cutover, a generated database probe must be derived from the owner-only candidate manifest rather than import application models while connected to production. Inside one server-attested read-only PostgreSQL transaction it must bind the managed physical schema and exactly match every candidate model table, column, PostgreSQL type identity, nullability flag, and ordered primary key; unknown legacy objects fail closed unless an evidence-backed, versioned allowlist explicitly names them.
- Every deployment-created container must receive the complete versioned lifecycle label set at creation. Offline journal recovery is bound to the exact trusted-bundle digest and may only stop/rename/start the precise recorded containers, quarantining rather than deleting the candidate. After online candidate qualification, orphan cleanup must validate the complete batch twice before its first deletion and agree on name, immutable container/image IDs, revision, run ID, role, command, environment class, and hardened Docker topology. Only the running official container is protected; a stopped rollback backup is retained for one release and removed at the next qualified deployment. One invalid object aborts before deletion begins; the delete phase uses full immutable IDs and is explicitly idempotent rather than transactionally atomic.
- Required UI automation must use the one shared application factory and inherit `XAgeUITestCase`; that base owns launch/terminate and automatically audits every launch in teardown. The source-wide `XCUIApplication` type/constructor and `.launch`/`.terminate` token counts are exact, so `.init`, contextual initialization, method references, nested helpers, or lifecycle calls outside the base fail. UI tests must not call the public internet, production accounts, a real model, HealthKit, notification permission, or an uncontrolled path monitor. External dependencies need an exact opt-in, `#if DEBUG` deterministic transport; unknown requests must fail closed and every launch must prove `intercepted>0;unhandled=0`. Release must keep production behavior, and live-network/system behavior must be tested separately.
- Continuous chat must satisfy `UX-CHAT-QUIESCENCE-001`: message, sending, thinking, keyboard, and upload state changes may not enqueue overlapping animated bottom-scroll operations, and both the XAGE and legacy message stacks must retain one ordered bottom-anchor sentinel wired only through synchronous, animation-disabled `ChatAutoScroll`. Return inserts a newline and must never send or dismiss the keyboard; the paper-plane button is the only typed-draft send action. A typed send must synchronously capture the immutable draft, release focus, and dismiss the keyboard before any asynchronous task starts; starter prompts, retries, consent continuations, report-upload follow-ups, and every other outbound sibling entry point must use their audited dismissal/wiring rather than a direct or aliased bypass. Thinking and upload status must both consume the shared `ChatProgressIndicator`, which is static only in explicit Debug UI automation and remains a normal `ProgressView` otherwise. Release and UI automation must not fork anchors, message/Markdown consumers, accessibility trees, outbound wiring, lifecycle values, or terminal behavior; the indicator's automation-static rendering is the sole audited visual-state exception. Visually plain text must expose a plain accessibility tree; Markdown supported by the current inline renderer must keep its prior visual formatting and marker-free reading order, while every rendered link remains a real accessible and hittable `Link` action. Deterministic UI tests must preserve multiline input, assert the app-owned terminal lifecycle (`idle`, exact message count, latest assistant, focus false, no thinking) before querying the keyboard, and verify the exact user/assistant text. The tools policy test must audit the repository-wide SwiftUI/UIKit scroll APIs, `UIScrollView`/`ScrollPosition`/`ScrollViewProxy`/`ChatAutoScroll`, transaction and `onChange` identifier inventories; lock the exact observers/helpers, anchors and order, every send path, lifecycle probe/live wiring, both progress consumers, keyboard installer, real Markdown consumers, accessibility replacement/link tree, complete root consumers, compiler-conditional blocks, shared UI wait/network-audit base, and terminal/multiline UI assertions; and reject all 68 current direct, animated, queued, overloaded, shadowed, bridged, out-of-surface, hard-coded-state, removed-dismiss, Return-submit, consumer-bypass, Release/automation-divergence, dead-audit, always-true-wait, duplicate-AX-tree, or inaccessible-link mutations. Simulator XCTest does not replace real-device VoiceOver rotor, reading-order, or activation sign-off.
- Production Swift may contain exactly one `URLSession` constructor: `APIService.trustedSession` configured by `APIService.makeSessionConfiguration()`. Requests inside `APIService` must use explicit `self.trustedSession`; every request outside it must use explicit `APIService.shared.trustedSession`; UI tests may not issue requests themselves. Source-wide token identities also pin `NWPathMonitor`, `HKHealthStore`, and `UNUserNotificationCenter.current` to their one audited UI-safe owner/factory. `URLSession.shared`, inferred/contextual sessions, a second constructor or alias, tuple/pattern shadowing or redeclaration of `APIService`/`trustedSession`, string-interpolation/raw-regex/escaped-identifier bypasses, contextual/direct system constructors, stored system factories, `Data`/`NSData`/`String(contentsOf:)`, `AsyncImage`, WebKit, direct Network/CFNetwork/POSIX APIs or function aliases are gate failures. Local URL reads must use `LocalFileDataLoader`, whose only direct Foundation read first requires `url.isFileURL`. This is a fail-closed static boundary for controlled source, not an OS outbound firewall; every new transport or system owner needs matching negative tests before use.
- Changed-path detection must include additions, modifications, deletions, copies, and both sides of renames. Both tracked and untracked regular files must pass whitespace validation. A test declaration, skip, or constant-true assertion is not evidence of a behavior regression test.
- Every file and directory under the pinned app/unit/UI source roots must be a real regular file/directory rather than a symlink; this includes Swift, Info.plist, entitlements, privacy manifests, assets and localization resources. The repository, source roots, project, shared scheme, and every pinned ancestor path must also be real. The PBX parser masks comments, tokenizes quoted strings, and requires critical OpenStep keys exactly once before validating the real object graph. The three pinned native targets must keep their exact configuration lists, build phases, rules and dependencies; every Sources entry must resolve build-file → fileRef → group path to the exact filesystem source set, framework phases must remain empty, and Swift package objects/references are forbidden. Project comments, strings, filenames, or section labels are not evidence. Fake comments/strings, duplicate keys, swapped references/targets, duplicate or hidden source phases, shell-script phases, missing/extra sources, and symlinked ancestors/files are gate failures.
- The project must retain the exact eight Debug/Release build configurations, four ordered configuration lists with Release defaults, three target configuration-list bindings, and one script-free shared scheme whose Build/Test/Launch/Profile/Archive actions point to the pinned targets and configurations. Static project settings forbid source exclusion/inclusion, compiler/linker injection, bridging headers, Swift include paths, library/framework search paths, testability or Release debug defines. Before every device archive, `xcodebuild -showBuildSettings -json` must independently prove the effective Release SDK/platform/architecture/configuration, compiler mode, entitlements, Info.plist, product identity and version/build, reject the same linker/bridging/compiler overrides, and allow only the generated target build directory in effective header/library/framework search paths.
- Every iOS source, project, configuration, test-support, quality-gate, or release change must create a fresh unsigned Release archive for `generic/platform=iOS` and pass `tools/verify_release_bundle.py` against that device archive. The verifier must reject a non-regular, symlinked, non-executable, ASCII, FAT, x86_64, dylib, Simulator, malformed, or otherwise non-device main executable; the accepted binary is a thin little-endian 64-bit arm64 `MH_EXECUTE` for iOS device platform 2 (or the legacy iPhoneOS load command). A Simulator-only Release build does not prove the device/TestFlight binary boundary.
- `TEST-SUITE-INTEGRITY-001` makes runtime IDs, parameter cases, skips, and obvious assertion deletion/constant replacement fail closed. Static automation cannot prove that every changed assertion preserves the same semantic strength; until an independent reviewer exists, this remains an explicit trust boundary and must not be described as impossible to weaken.

## Conservative XAGE rule

The XAGE Home surface is permanently governed by the exact ordered seven-file source manifest in `quality/swift_source_manifest.json`: contracts, root shell, data dashboard, conversation, healthspan, settings, and shared components. Recombining those responsibilities into a monolith, omitting a role, adding an unlisted `XAge*.swift` file, or raising a per-role/aggregate architecture limit is forbidden. Any modification to the governed source set is conservatively treated as UI + interaction + AI client + Health client + account risk; `XAgeDataDashboard.swift` is already near its cap, so further growth must be extracted at a real responsibility boundary without weakening the manifest or shared state behavior.

The UI prompt loop in `XAgeHighIntensityContextUITests` proves input and shell interactions only. It must not be cited as proof that AI answer content, subject isolation, routing, citations, or safety are correct. Those claims require deterministic backend/client assertions or a controlled end-to-end evaluation that verifies the final assistant response.

## Release gate

TestFlight must not be archived or uploaded through an ad-hoc direct command. Required order:

There is no diagnostic exception. Never run a signed archive or `xcodebuild -exportArchive` for an ineligible/currently uploaded build; compatibility work must use synthetic fixtures until an eligible candidate has passed `release` and `assert-release`. The fresh unsigned generic-device archive required by the ordinary regression gate is the only pre-release archive exception.

1. Work on a feature branch based on `main`, push it, and merge it into `main` through a pull request. `main` is the only development, PR, CI, deployment, and release-candidate branch. `XAGE` is a protected, locked, read-only legacy branch and must never receive another delivery.
2. Check out/pull the exact post-merge `main` candidate so the worktree is clean and `HEAD` equals `origin/main`; wait for the push-triggered GitHub Actions `quality-gate` to succeed on that exact SHA. A manual-dispatch run, different SHA/branch/workflow/check app, skipped run, cached claim, or XAGE run is invalid.
3. Before collecting sign-offs, require the candidate `CURRENT_PROJECT_VERSION` to be strictly greater than `release_gate.latest_uploaded_build`. Build `17` is already uploaded and is therefore ineligible; the next candidate is at least build `18`. Complete all five real-device/controlled sign-offs from `quality/release_signoffs.example.json` only against that new candidate and save the HEAD/tree/registry-bound result as ignored `.quality/release_signoffs.json`. Every item must be `passed`, bind `app_version` and `app_build` to the candidate's unique `MARKETING_VERSION` and `CURRENT_PROJECT_VERSION`, and include a non-placeholder tester, timezone-aware test time, device/system/build environment, at least two repeatable steps/observations, and a redacted regular local evidence file under `.quality/evidence/` with its real SHA-256. URLs, symlinks, missing files, stale/wrong app builds, or digest mismatches fail closed. All evidence must remain within the release lifetime.
4. Run `/usr/bin/python3 -I tools/run_regression_gate.py release`; ordinary `python3`, a non-isolated interpreter, user startup configuration, or a user-writable interpreter cannot generate release evidence. The gate validates the sign-off file, the official repository default branch and candidate tip are both `main`, the merged PR associated with exact `HEAD` targets `main`, `main` is protected and writable only through PR while `XAGE` remains protected and locked, the exact-SHA main push workflow/check, trusted root-owned/non-writable Xcode `26.3` (`17C529`) toolchain, bound backend runtime/JUnit, and all mandatory local commands before and after the gate. Release evidence schema `5` binds these identities and the candidate app version/build.
5. Immediately before archive/export, run `/usr/bin/python3 -I tools/run_regression_gate.py assert-release`; cached evidence alone is insufficient because the command rechecks GitHub, both branch protections, sign-offs, app version/build, gate/backend/Xcode runtimes, JUnit, tree and registry live state.
6. Use `scripts/release_testflight.sh --archive-only` or `scripts/release_testflight.sh --upload`; do not call `xcodebuild -exportArchive` or an uploader directly. Both modes archive, export exactly one IPA in a random owner-only directory, and create a random owner-only, single-hard-link, read-only candidate snapshot. Before extraction they bind the ZIP local headers/data and central-directory entries one-to-one and contiguously through the sole final EOCD, reject prefixes, gaps, comments, trailing bytes, unapproved local/central extra forms, identity mismatches and unreferenced metadata, and parse the exact signed data-descriptor form emitted by pinned Xcode. They then scan the raw IPA and every member—not only `Payload/*.app`—for unsafe/colliding paths, links or special entries, abnormal expansion, sensitive filenames, SQLite/JWK/OpenSSH/private-key structures and PEM/DER/PKCS#8 material. Bounded binary scans fail closed on an oversized JSON/DER candidate or a whitespace-only prefix. They safely inspect the distribution `.app`, require one trusted Apple provisioning CMS signer before decoding the profile, verify version/build, production API, device Mach-O, signing leaf certificate against that profile, HealthKit entitlements, source/copy hashes, IPA SHA-256 and distribution CDHash. `--archive-only` stops after verification. `--upload` may then use only the pinned Xcode `altool` with that same read-only snapshot path, after a final inode/device/link-count/mode/size/hash recheck, using either complete API key ID/issuer metadata or a username plus Keychain-item reference; no credential value belongs in code, docs, memory, logs, or arguments supplied by a wrapper.
7. If source, tests, build settings, remote tip, registry, sign-offs, branch protection, runtime identity, or `HEAD` changes, the evidence is invalid and the full release gate must run again. After a successful upload of build `N`, advance `latest_uploaded_build` to `N` through the next protected PR before another release.

`BRANCH-CANONICAL-001` is not satisfied by CI filters alone. The official repository default branch must be `main`; `main` must require the exact GitHub Actions `quality-gate`, PR delivery with no bypass, administrator enforcement, and no force-push/deletion, while remaining unlocked for protected PR merges. `XAGE` must retain the same protection and additionally expose `lock_branch=true` and `allow_fork_syncing=false`. Approval count is pinned to zero while the repository has only its owner and no independent reviewer; this reduces accidental delivery but does not prevent that owner from weakening workflow, gate, tests, and constants together. Adding a real collaborator must strengthen protection to one approval plus last-push approval and should place quality/release paths behind independent CODEOWNERS, a required workflow/ruleset, or an approved release environment.

A sign-off file path and SHA-256 prove only that the referenced local bytes are present and unchanged. They do not authenticate the tester, prove the steps occurred, or make a publisher's self-attestation independent evidence. The publisher must not fill the five sign-offs as their own proof; release requires genuine human/QA evidence whose identity and observations can be independently checked. Automation and evidence integrity reduce accidental regression risk, but neither is a substitute for this external trust boundary.

For a HealthKit release, Simulator results never replace real-device authorization, foreground sync, background observer, and source/account verification. For an AI release, typing prompts without asserting the returned answer never counts as AI-content validation.
