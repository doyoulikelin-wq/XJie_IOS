## Conservative XAGE rule

The XAGE Home surface is permanently governed by the exact ordered seven-file source manifest in `quality/swift_source_manifest.json`: contracts, root shell, data dashboard, conversation, healthspan, settings, and shared components. Recombining those responsibilities into a monolith, omitting a role, adding an unlisted `XAge*.swift` file, or raising a per-role/aggregate architecture limit is forbidden. Any modification to the governed source set is conservatively treated as UI + interaction + AI client + Health client + account risk; `XAgeDataDashboard.swift` is already near its cap, so further growth must be extracted at a real responsibility boundary without weakening the manifest or shared state behavior.

The UI prompt loop in `XAgeHighIntensityContextUITests` proves input and shell interactions only. It must not be cited as proof that AI answer content, subject isolation, routing, citations, or safety are correct. Those claims require deterministic backend/client assertions or a controlled end-to-end evaluation that verifies the final assistant response.

## Release gate

TestFlight must not be archived or uploaded through an ad-hoc direct command. The TestFlight contract has two distinct stages: an internal candidate upload, followed by post-upload qualification of the exact Apple-accepted build. Upload success alone is never final qualification, and neither TestFlight stage weakens the separate schema `5` final-release gate.

There is no diagnostic exception. Never run a signed archive or `xcodebuild -exportArchive` for an ineligible/currently uploaded build; compatibility work must use synthetic fixtures until an eligible candidate has passed `internal-testflight` and `assert-internal-testflight`. The fresh unsigned generic-device archive required by the candidate gate is the only pre-upload archive exception.

1. Work on a feature branch based on `main`, push it, and merge it into `main` through a pull request. `main` is the only development, PR, CI, deployment, and release-candidate branch. `XAGE` is a protected, locked, read-only legacy branch and must never receive another delivery.
2. Check out/pull the exact post-merge `main` candidate so the worktree is clean and `HEAD` equals `origin/main`; wait for the push-triggered GitHub Actions `quality-gate` to succeed on that exact SHA. A manual-dispatch run, different SHA/branch/workflow/check app, skipped run, cached claim, or XAGE run is invalid.
3. Require the candidate `CURRENT_PROJECT_VERSION` to be strictly greater than `release_gate.latest_uploaded_build`. Build `18` is already uploaded and is therefore ineligible; the next upload candidate is at least build `19`. The tracked build `18` `pending_internal_candidate` must first be successfully qualified or explicitly rejected/retired through a protected registry change; `internal-testflight` fails closed while any pending candidate remains. The five real-device/controlled sign-offs are deliberately not pre-upload inputs: at this stage a new candidate is eligible only for restricted internal TestFlight distribution.
4. Run `/usr/bin/python3 -I tools/run_regression_gate.py internal-testflight`; ordinary `python3`, a non-isolated interpreter, user startup configuration, or a user-writable interpreter cannot generate its evidence. The gate validates the complete deterministic automation, official `main`, merged PR and exact-SHA push check, distinct main/XAGE protections, trusted Xcode `26.3` (`17C529`), bound backend runtime/JUnit and all mandatory local commands. It writes only schema `1`, phase `internal_testflight_upload` evidence to `.quality/internal_testflight_gate.json`, without `manual_signoffs` or any final-qualification claim.
5. Immediately before archive/export, run `/usr/bin/python3 -I tools/run_regression_gate.py assert-internal-testflight`; cached evidence alone is insufficient because the command rechecks GitHub, both branch protections, app version/build, gate/backend/Xcode runtimes, JUnit, tree, registry live state and absence of another pending candidate.
6. Use `scripts/release_testflight.sh --archive-only` or `scripts/release_testflight.sh --upload`; do not call `xcodebuild -exportArchive` or an uploader directly. Both modes archive, export exactly one IPA in a random owner-only directory, and create a random owner-only, single-hard-link, read-only candidate snapshot. Before extraction they bind the ZIP local headers/data and central-directory entries one-to-one and contiguously through the sole final EOCD, reject prefixes, gaps, comments, trailing bytes, unapproved local/central extra forms, identity mismatches and unreferenced metadata, and parse the exact signed data-descriptor form emitted by pinned Xcode. They then scan the raw IPA and every member—not only `Payload/*.app`—for unsafe/colliding paths, links or special entries, abnormal expansion, sensitive filenames, SQLite/JWK/OpenSSH/private-key structures and PEM/DER/PKCS#8 material. Bounded binary scans fail closed on an oversized JSON/DER candidate or a whitespace-only prefix. They safely inspect the distribution `.app`, require one trusted Apple provisioning CMS signer before decoding the profile, verify version/build, production API, device Mach-O, signing leaf certificate against that profile, HealthKit entitlements, source/copy hashes, IPA SHA-256 and distribution CDHash. Once finalized, the snapshot parent is locked to mode `0500`; its realpath/device/inode/link-count/mode/owner and the IPA realpath/device/inode/link-count/mode/size/hash are bound and rechecked before and after the uploader. This prevents routine or accidental path replacement but is not isolation from an actively malicious same-UID publisher that can chmod, ptrace, or control the release process. `--archive-only` stops after verification. `--upload` may then use only the pinned Xcode `altool` with that bounded-trust snapshot path, using either complete API key ID/issuer metadata or a username plus Keychain-item reference; no credential value belongs in code, docs, memory, logs, or arguments supplied by a wrapper.
7. `scripts/release_testflight.sh` must invoke `assert-internal-testflight` exactly three times: before archive, after signed archive verification, and after the verified distribution IPA exists. Immediately before calling the uploader, `--upload` atomically creates `.quality/internal_testflight_upload_attempts/<version>-<build>.json`; the tombstone is retained after success, failure, or timeout, so the same publisher machine must never retry that version/build. Pinned Xcode `altool` stdout JSON and stderr banner must be captured in separate owner-only `0600` temporary files: validate and hash stdout only, never merge stderr with `2>&1`, never print raw uploader output, and delete both files in the EXIT cleanup. Only after the stdout JSON contains an empty `product-errors` list, a non-empty `success-message`, no reported `errors`, and the same read-only IPA passes the final SHA-256/CDHash/profile/bundle recheck may the script atomically create `.quality/internal_testflight_upload_receipts/<version>-<build>.json`. The local tombstone cannot coordinate independent machines: Apple build-number uniqueness plus a single authorized publisher remain the cross-machine boundary. Advance `latest_uploaded_build` and record that receipt as `pending_internal_candidate` through a protected PR before another upload.
8. Build `18` is a special historical pending candidate uploaded directly by Xcode with managed remote signing. Its tracked receipt identifier is `xcode-distribution:<distribution_identifier>` and proves the Apple distribution identifier, app/build, certificate and upload/archive log digests, but it permanently retains `ipa_sha256=null`, `distribution_cdhash=null`, and `external_promotion_allowed=false`. It may pass internal TestFlight qualification as the recorded historical candidate, but it can never become externally promotable, be cited as proof that a locally verified IPA was the exact uploaded package, or serve as precedent for omitting strict same-IPA verification on build `19+`.
9. Install the Apple-accepted build from TestFlight, not Xcode or a locally exported IPA, and complete all five `post_upload_signoffs` from `quality/testflight_signoffs.example.json`: real iPhone HealthKit, Apple Watch/background sync, third-party Chinese keyboard, large text/VoiceOver, and controlled final AI answer. Save the completed file as `.quality/testflight_signoffs.json` and evidence under `.quality/evidence/`. The top level and every item must bind the same `pending_candidate_sha256` and `upload_receipt_identifier`; future verified uploads use `altool-result-sha256:<upload_result_sha256>`, while historical build `18` uses `xcode-distribution:<distribution_identifier>`. The top level must bind the exact candidate `HEAD`/tree/registry blob, and every item must be `passed`, use `installation_source=TestFlight`, bind the exact candidate version/build, identify a non-placeholder tester and environment, include a timezone-aware `tested_at` strictly later than `uploaded_at` and at least two repeatable steps/observations, and reference a redacted regular local evidence file with its real SHA-256. Post-upload TestFlight evidence has its own pinned seven-day freshness window so the planned three-day natural-use test remains valid; final schema `5` release sign-offs retain the separate 24-hour window.
10. Run `/usr/bin/python3 -I tools/run_regression_gate.py qualify-testflight`. It validates the tracked pending receipt and all five TestFlight-origin sign-offs, then atomically writes `.quality/testflight_qualifications/<version>-<build>.json`. External promotion is permitted only when that qualification records `external_promotion_allowed=true`, which requires a future `verified_local_ipa_altool` receipt with valid IPA SHA-256 and distribution CDHash; historical build `18` remains internal-only even after qualification. If qualification fails, preserve the evidence, explicitly reject/retire the pending candidate through a protected registry change, fix through a new main-based PR, increment `CURRENT_PROJECT_VERSION` above the uploaded build, and repeat both TestFlight stages. Whether qualification passes or the candidate is rejected, resolve the tracked pending state before uploading the next build.
11. The existing `/usr/bin/python3 -I tools/run_regression_gate.py release` and `assert-release` remain a separate final-release contract. Create `.quality/release_signoffs.json` only from `quality/release_signoffs.example.json`; these commands accept only schema `5` `.quality/release_gate.json` bound to the current candidate and the registry's `manual_signoffs`. `.quality/internal_testflight_gate.json`, an upload receipt, `.quality/testflight_signoffs.json`, or `.quality/testflight_qualifications/<version>-<build>.json` must never be converted, reused or described as schema `5` final-release evidence.

If source, tests, build settings, remote tip, branch protection, runtime identity or candidate `HEAD` changes before upload, the internal evidence is invalid and `internal-testflight` must run again. After upload, the candidate `HEAD`/tree/version/build and receipt identity are immutable; the protected registry PR that advances `latest_uploaded_build` and records or resolves `pending_internal_candidate` is expected, but it must not rewrite the uploaded candidate identity. Product or candidate-byte changes require a new build number.

`BRANCH-CANONICAL-001` is not satisfied by CI filters alone. The official repository default branch must be `main`; `main` must require the exact GitHub Actions `quality-gate`, PR delivery with no bypass, administrator enforcement, and no force-push/deletion, while remaining unlocked for protected PR merges. `XAGE` must retain the same protection and additionally expose `lock_branch=true` and `allow_fork_syncing=false`. Approval count is pinned to zero while the repository has only its owner and no independent reviewer; this reduces accidental delivery but does not prevent that owner from weakening workflow, gate, tests, and constants together. Adding a real collaborator must strengthen protection to one approval plus last-push approval and should place quality/release paths behind independent CODEOWNERS, a required workflow/ruleset, or an approved release environment.

A sign-off file path and SHA-256 prove only that the referenced local bytes are present and unchanged. They do not authenticate the tester, prove the steps occurred, or make a publisher's self-attestation independent evidence. The publisher must not fill the five sign-offs as their own proof; final qualification requires genuine human/QA evidence whose identity, TestFlight receipt and observations can be independently checked. Automation and evidence integrity reduce accidental regression risk, but neither is a substitute for this external trust boundary.

For a HealthKit release, Simulator results never replace real-device authorization, foreground sync, background observer, and source/account verification. For an AI release, typing prompts without asserting the returned answer never counts as AI-content validation.
